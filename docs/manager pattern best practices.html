The Manager Pattern: Deep Dive

The manager pattern is a way to separate business logic and coordination from UI presentation. Think of it as the "conductor" of an orchestra - it doesn't play the instruments (UI), but it coordinates when and how they play.

Core Principles

1. Clear Separation of Concerns

Manager Responsibilities:

typescript
// ✅ GOOD - Manager controls WHEN and WHERE
manager.show('buffer', x, y);        // Decides to show
manager.hide();                       // Decides to hide
manager.handleSelect(macro);          // Business logic (text replacement)
manager.destroy();                    // Lifecycle management

Component Responsibilities:

typescript
// ✅ GOOD - Component controls HOW IT LOOKS
const filteredMacros = useMemo(...); // UI filtering
const navigation = useNavigation();   // UI state
useKeyboardNavigation(...);          // UI interactions

Anti-pattern:

typescript
// ❌ BAD - Manager trying to control UI internals
manager.selectNext();     // Component should handle this
manager.highlightIndex(); // Component should handle this
manager.scrollToItem();   // Component should handle this

2. Single Source of Truth

Problem in your original code:

typescript
// Manager
const getMatchingMacros = (buffer: string): Macro[] => {
  return macros.filter(macro => 
    macro.command.toLowerCase().startsWith(buffer.toLowerCase())
  ).slice(0, 5);
};

// Component (DUPLICATE!)
const filteredMacros = macros.filter(macro => 
  macro.command.toLowerCase().startsWith(buffer.toLowerCase())
).slice(0, 5);

Solution - Pick ONE place:

Option A: Manager filters (good for complex logic)

typescript
// Manager
const filteredMacros = getMatchingMacros(buffer);
renderer.render(createElement(Component, { macros: filteredMacros }));

// Component just displays
function Component({ macros }) {
  return macros.map(m => <div>{m.command}</div>);
}

Option B: Component filters (good for UI-driven logic)

typescript
// Manager passes raw data
renderer.render(createElement(Component, { 
  allMacros: macros,
  filterBuffer: buffer 
}));

// Component filters
function Component({ allMacros, filterBuffer }) {
  const filtered = useMemo(() => 
    allMacros.filter(m => m.command.includes(filterBuffer))
  , [allMacros, filterBuffer]);
}

3. Data Flows Down, Events Flow Up

typescript
// ✅ GOOD - Clear unidirectional flow
┌─────────────────────────────────────┐
│           MANAGER                    │
│  (Business Logic & Coordination)     │
│                                      │
│  - Decides when to show              │
│  - Provides data                     │
│  - Handles selection business logic  │
└──────────────┬──────────────────────┘
               │ Props ↓
               │ (data, callbacks)
               │
┌──────────────▼──────────────────────┐
│          COMPONENT                   │
│  (Presentation & UI State)           │
│                                      │
│  - Renders UI                        │
│  - Manages navigation state          │
│  - Handles keyboard events           │
└──────────────┬──────────────────────┘
               │ Callbacks ↑
               │ (onSelect, onClose)
               │
┌──────────────▼──────────────────────┐
│           MANAGER                    │
│  Executes business logic             │
└─────────────────────────────────────┘

Common Anti-Patterns to Avoid

❌ Anti-Pattern #1: State Synchronization

typescript
// BAD - Trying to keep manager and component in sync
const manager = {
  currentSelectedIndex: 0,
  render() {
    createElement(Component, {
      selectedIndex: this.currentSelectedIndex,
      onSelectedIndexChange: (index) => {
        this.currentSelectedIndex = index; // Circular dependency!
      }
    });
  }
};

Why it's bad:

  • Creates circular dependencies
  • Two sources of truth that can get out of sync
  • Makes debugging hard ("which state is correct?")

Solution:

typescript
// GOOD - Component owns its UI state
const manager = {
  render() {
    createElement(Component, {
      onSelect: (item) => this.handleSelect(item)
      // Component manages its own selectedIndex internally
    });
  }
};

❌ Anti-Pattern #2: Manager Controlling UI Details

typescript
// BAD - Manager knows too much about UI
const manager = {
  selectNext() {
    if (this.selectedIndex < this.items.length - 1) {
      this.selectedIndex++;
      this.scrollItemIntoView(this.selectedIndex);
      this.updateHighlight(this.selectedIndex);
    }
  }
};

Why it's bad:

  • Manager is concerned with scrolling, highlighting (UI details)
  • Hard to change UI without changing manager
  • Difficult to test manager without UI

Solution:

typescript
// GOOD - Manager only handles business events
const manager = {
  show(items) {
    // Manager just provides data and callbacks
    createElement(Component, {
      items,
      onSelect: (item) => this.performBusinessLogic(item)
    });
  }
};

// Component handles UI interactions
function Component({ items, onSelect }) {
  const [selectedIndex, setSelectedIndex] = useState(0);
  
  const handleKeyDown = (e) => {
    if (e.key === 'ArrowDown') {
      setSelectedIndex(prev => Math.min(prev + 1, items.length - 1));
    }
    if (e.key === 'Enter') {
      onSelect(items[selectedIndex]);
    }
  };
}

❌ Anti-Pattern #3: Exposing Internal State

typescript
// BAD - Exposing getters/setters for everything
interface Manager {
  getBuffer(): string;
  setBuffer(buffer: string): void;
  getSelectedIndex(): number;
  setSelectedIndex(index: number): void;
  getCursorPosition(): Position;
  setCursorPosition(pos: Position): void;
}

Why it's bad:

  • Other code can manipulate internal state
  • Breaks encapsulation
  • Makes it hard to maintain invariants

Solution:

typescript
// GOOD - Only expose necessary operations
interface Manager {
  show(buffer: string, x?: number, y?: number): void;
  showAll(x?: number, y?: number): void;
  hide(): void;
  isVisible(): boolean;
  updateMacros(macros: Macro[]): void;
  destroy(): void;
}

Real-World Examples

Example 1: Modal Manager

typescript
// ✅ GOOD Modal Manager
function createModalManager() {
  let currentModal: ModalData | null = null;
  
  return {
    // Manager decides WHEN to show and WHAT data
    showConfirm(title: string, message: string) {
      currentModal = { type: 'confirm', title, message };
      render();
    },
    
    showAlert(message: string) {
      currentModal = { type: 'alert', message };
      render();
    },
    
    // Manager handles business logic
    handleConfirm() {
      const result = currentModal?.onConfirm?.();
      this.close();
      return result;
    },
    
    close() {
      currentModal = null;
      render();
    }
  };
}

// Component handles HOW it looks
function Modal({ data, onConfirm, onCancel }) {
  const [isAnimating, setIsAnimating] = useState(false);
  
  // Component owns animation state
  useEffect(() => {
    setIsAnimating(true);
  }, []);
  
  return (
    <div className={isAnimating ? 'fade-in' : ''}>
      <h1>{data.title}</h1>
      <p>{data.message}</p>
      <button onClick={onConfirm}>OK</button>
      <button onClick={onCancel}>Cancel</button>
    </div>
  );
}

Example 2: Notification Manager

typescript
// ✅ GOOD Notification Manager
function createNotificationManager() {
  const notifications: Notification[] = [];
  
  return {
    // Manager handles business logic
    success(message: string) {
      const id = generateId();
      notifications.push({ id, type: 'success', message });
      render();
      
      // Auto-dismiss after 3 seconds
      setTimeout(() => this.dismiss(id), 3000);
    },
    
    error(message: string) {
      const id = generateId();
      notifications.push({ id, type: 'error', message });
      render();
      // Errors don't auto-dismiss
    },
    
    dismiss(id: string) {
      const index = notifications.findIndex(n => n.id === id);
      if (index !== -1) {
        notifications.splice(index, 1);
        render();
      }
    }
  };
}

// Component handles presentation
function NotificationList({ notifications, onDismiss }) {
  return (
    <div className="notification-container">
      {notifications.map(notif => (
        <Notification
          key={notif.id}
          notification={notif}
          onDismiss={() => onDismiss(notif.id)}
        />
      ))}
    </div>
  );
}

// Each notification handles its own animation
function Notification({ notification, onDismiss }) {
  const [isVisible, setIsVisible] = useState(false);
  
  useEffect(() => {
    setIsVisible(true);
  }, []);
  
  const handleDismiss = () => {
    setIsVisible(false);
    setTimeout(onDismiss, 300); // Wait for animation
  };
  
  return (
    <div className={`notification ${isVisible ? 'show' : 'hide'}`}>
      {notification.message}
      <button onClick={handleDismiss}>×</button>
    </div>
  );
}

Testing Best Practices

Manager Tests - Focus on Business Logic

typescript
describe('Manager', () => {
  test('shows overlay when called', () => {
    const manager = createManager();
    manager.show('buffer', 100, 200);
    
    expect(manager.isVisible()).toBe(true);
    // Test that correct props were passed to component
  });
  
  test('handles text replacement correctly', () => {
    const manager = createManager();
    const mockElement = createMockElement('test');
    
    manager.show('test', 100, 200);
    manager.handleSelect(macro);
    
    expect(replaceText).toHaveBeenCalledWith(
      mockElement, macro, 0, 4
    );
  });
});

Component Tests - Focus on UI Behavior

typescript
describe('Component', () => {
  test('filters macros correctly', () => {
    render(<Component macros={macros} filterBuffer="test" />);
    
    expect(screen.getByText('test-macro')).toBeInTheDocument();
    expect(screen.queryByText('other')).not.toBeInTheDocument();
  });
  
  test('handles keyboard navigation', () => {
    render(<Component macros={macros} onSelect={onSelect} />);
    
    fireEvent.keyDown(screen.getByRole('listbox'), { key: 'ArrowDown' });
    fireEvent.keyDown(screen.getByRole('listbox'), { key: 'Enter' });
    
    expect(onSelect).toHaveBeenCalledWith(macros[1]);
  });
});

When to Use the Manager Pattern

✅ Good Use Cases:

  1. Overlays/Popups - You need to coordinate positioning, showing/hiding
  2. Global state coordination - Multiple components need to react to same events
  3. Complex lifecycle - Setup, teardown, resource management
  4. Business logic isolation - Text replacement, API calls, data transformation
  5. Cross-cutting concerns - Keyboard shortcuts that affect multiple components

❌ When NOT to Use:

  1. Simple components - Just use React state
  2. Purely presentational - No business logic needed
  3. Single responsibility - Component can handle everything itself
  4. No coordination needed - No external state to manage

Summary: The Golden Rules

  1. Manager controls WHEN, Component controls HOW
  2. Data flows down, events flow up
  3. One source of truth for each piece of state
  4. Manager doesn't know about UI details (scrolling, highlighting, etc.)
  5. Component doesn't know about business logic (where data comes from, what happens on select)
  6. If you're syncing state between manager and component, you're doing it wrong
  7. Test business logic in manager tests, UI behavior in component tests
