Fantastic point! Naming is SO important for understanding. Let's explore alternatives:

Current Names (Abstract/Generic):

Actions      → What does this DO?
Coordinators → Coordinates what?
Managers     → Manages what?
Detector     → This one is good! (Detects macros)

You're right - these are programmer jargon, not domain language.

Alternative Naming Schemes:

Option 1: Role-Based (What They Do)

Actions      → EventHandlers  or  Callbacks
Coordinators → Responders  or  Handlers
Managers     → Controllers  or  Widgets
Detector     → MacroDetector (keep it!)

Example:

typescript
interface MacroEventHandlers {
  onMacroCommitted(id: string): void
}

function createSuggestionsResponder(): MacroEventHandlers { ... }
function createStatisticsResponder(): MacroEventHandlers { ... }

function createOverlayController() { ... }

Option 2: Domain-Based (What They Represent)

Actions      → MacroEvents
Coordinators → MacroListeners  or  MacroObservers
Managers     → OverlayView  or  UIComponent
Detector     → MacroScanner

Example:

typescript
interface MacroEvents {
  onMacroCommitted(id: string): void
}

function createSuggestionsListener(): MacroEvents { ... }
function createStatisticsObserver(): MacroEvents { ... }

function createSearchOverlayView() { ... }

Option 3: Functional (Verb-Based)

Actions      → WhenMacroHappens
Coordinators → RespondToMacro
Managers     → ShowOverlay / TrackStatistics
Detector     → DetectMacros

Example:

typescript
interface WhenMacroHappens {
  onCommitted(id: string): void
  onDetected(buffer: string): void
}

function respondWithSuggestions(): WhenMacroHappens { ... }
function respondWithStatistics(): WhenMacroHappens { ... }

Option 4: Architectural (Clear Layers)

Layer 1: Core Logic
  - MacroDetector

Layer 2: Business Logic  
  - SuggestionsPolicy    ← "What's the policy for showing suggestions?"
  - StatisticsPolicy     ← "What's the policy for tracking?"
  
Layer 3: Infrastructure
  - SearchOverlay
  - SuggestionsOverlay
  - StatisticsStorage

Example:

typescript
interface DetectorPolicy {
  whenMacroCommitted(id: string): void
  whenDetectionStarted(buffer: string): void
}

function createSuggestionsPolicy(): DetectorPolicy {
  return {
    whenMacroCommitted(id) {
      searchOverlay.hide()
    },
    whenDetectionStarted(buffer) {
      suggestionsOverlay.show(buffer)
    }
  }
}

Option 5: Simple English (No Jargon)

Actions      → MacroCallbacks
Coordinators → SuggestionsLogic / StatisticsLogic
Managers     → OverlayUI / SearchUI
Detector     → MacroDetector

Example:

typescript
interface MacroCallbacks {
  onMacroCommitted(id: string): void
}

function createSuggestionsLogic(): MacroCallbacks {
  return {
    onMacroCommitted(id) {
      overlayUI.hide()
    }
  }
}

const overlayUI = createSearchOverlay()
const suggestionsUI = createSuggestionsOverlay()

My Personal Favorite: Option 4 (Architectural)

Why?

typescript
// ============ Core: What the system does ============
const detector = createMacroDetector(policy)

// ============ Policy: Business rules ============
const suggestionsPolicy = createSuggestionsPolicy()
const statisticsPolicy = createStatisticsPolicy()
const combinedPolicy = combine(suggestionsPolicy, statisticsPolicy)

// ============ Infrastructure: How it's done ============
const searchOverlay = createSearchOverlay()
const suggestionsOverlay = createSuggestionsOverlay()
const statsStorage = createStatisticsStorage()

"Policy" clearly means: "These are the rules for how the system responds"

Your Statistics Feature with Better Names:

typescript
// ============ core/macroDetector.ts ============
function createMacroDetector(handlers: MacroEventHandlers) { ... }

// ============ logic/statisticsTracking.ts ============
function trackStatistics(): MacroEventHandlers {
  const stats = new Map()
  
  return {
    onMacroCommitted(id) {
      stats.set(id, (stats.get(id) || 0) + 1)
    },
    // ... other handlers
    
    // Extra methods
    getUsageCount(id: string) { return stats.get(id) || 0 },
    getAllStats() { return new Map(stats) }
  }
}

// ============ main.ts ============
const suggestionsHandler = handleSuggestionsDisplay()
const statisticsTracker = trackStatistics()

const combined = combineHandlers(suggestionsHandler, statisticsTracker)
const detector = createMacroDetector(combined)

Use Option 4 or Option 5 - they're the most intuitive.

Personally, I'd go with:

  • MacroEventHandlers instead of DetectorActions
  • handleX() or trackX() instead of createXCoordinator()
  • Keep Manager → just call them by what they manage: searchOverlay, statsStorage
